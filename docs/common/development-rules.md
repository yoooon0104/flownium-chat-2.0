# 개발 규칙

## 공통 개발 규칙

1. 코드 수정, 문서 수정, 커밋, 푸시, 실행 작업을 시작하기 전에 항상 이 문서를 먼저 확인한다.
2. 이 문서에 적힌 규칙과 실제 작업 방식이 어긋나면, 작업을 더 진행하기 전에 문서를 먼저 갱신한다.
3. 모든 MongoDB 모델은 `timestamps: true`를 사용한다.
4. 현재 MVP는 그룹채팅 기준이며, 1:1 `roomKey` 중복 방지 규칙은 이번 단계에서 사용하지 않는다.
5. `ChatRoom`은 메시지 저장 시 `lastMessage`, `lastMessageAt`를 함께 갱신한다.
6. REST/Socket 스펙이 바뀌면 관련 `docs/*.md`를 같은 작업에서 동기화한다.
7. REST와 Socket 모두 JWT 인증을 적용한다.
8. URL 하드코딩을 피하고 환경변수를 사용한다.
9. online 상태는 소켓 메모리 presence 맵 기준으로 계산한다.
10. 큰 기능 방향이 바뀌면 `docs/common/wbs.md`와 설계 문서를 함께 갱신한다.
11. 비동기 로직은 `async/await`를 기본으로 사용한다.
12. 중/대규모 작업은 변경안을 먼저 공유하고 리뷰 후 구현한다.
13. 코드 주석은 한글로 작성하고, 핵심 비즈니스 로직에는 의도/입출력 기준 설명을 남긴다.
14. 소스 수정이나 기능 추가를 할 때는 해당 흐름의 주석도 같은 작업에서 바로 반영한다. 주석을 나중에 몰아서 보강하지 않는다.
15. 레거시 DB 인덱스/스키마 이슈가 있으면 서버 시작 시점 보정 로직 또는 마이그레이션 계획을 문서화한다.
16. 운영 설정 이슈는 추측보다 로그와 실제 설정값을 먼저 확인한다.
17. 한글이 포함된 소스/문서 파일은 UTF-8로 저장한다. PowerShell 기본 인코딩으로 저장하지 않는다.
18. PowerShell로 파일을 쓸 때는 UTF-8 명시 저장 방식을 사용한다.
19. 문자열 치환으로 파일을 수정할 때는 exact string replace를 기본으로 쓰지 않는다. CRLF, 들여쓰기, 한글 주석, 템플릿 문자열 때문에 쉽게 실패하므로 먼저 실제 줄을 읽고 수정 방식을 결정한다.
20. 코드 수정은 가능하면 `apply_patch`를 우선 사용하고, 불가할 때만 정규식 기반 치환을 사용한다. heredoc 안의 템플릿 문자열이나 역따옴표는 이스케이프 문제를 만들기 쉬우므로 줄 배열 결합 같은 더 단순한 방식을 우선 검토한다.
21. 정규식이나 스크립트로 파일을 수정한 뒤에는 수정된 줄을 다시 읽어서 실제 반영 여부를 바로 확인한다. 빌드나 구문 체크만으로 반영 성공을 판단하지 않는다.
22. 문자열 치환이 두 번 이상 실패하면 같은 방식으로 반복하지 말고, 현재 파일 블록을 다시 읽고 수정 전략을 바꾼다.
23. 한글이 포함된 파일을 수정한 뒤에는 `Get-Content -Encoding utf8` 또는 동등한 방식으로 실제 한글이 정상 표시되는지 다시 확인한다. UTF-8 저장만 했다고 바로 완료로 판단하지 않는다.
24. 파일 수정 명령이 권한 부족으로 막히면 같은 방법만 반복하지 말고, 허용된 범위 안에서 `apply_patch`, UTF-8 명시 저장, 정규식 치환, 블록 단위 재작성처럼 더 안전한 대체 전략으로 재시도한다.
25. 부분 삽입으로 문서를 고칠 때는 섹션 경계 정규식이나 exact string replace를 그대로 믿지 않는다. 제목, 빈 줄, 코드블록 경계, CRLF/LF 차이 때문에 잘못된 위치에 삽입되거나 깨질 수 있으므로 현재 블록을 다시 읽고 범위를 확인한 뒤 수정한다.
26. 인코딩 문제가 발생한 파일은 깨진 줄만 부분 치환하지 말고, 필요하면 문서 전체 또는 해당 섹션 전체를 UTF-8로 다시 쓰는 방식을 우선 검토한다. 부분 삽입 상태에서 추가 수정하면 깨짐이 더 커질 수 있다.
27. 스크립트 수정 후에는 빌드/구문 체크 외에 실제 파일 내용을 다시 읽어서 제목, 섹션 경계, 한글 문장, 코드블록이 의도한 위치에 들어갔는지 확인한다.

## 배포/검증 규칙

1. 배포 전 `npm run lint`, `npm run build`를 반드시 통과시킨다.
2. 서버 변경 시 `node --check server/index.cjs`로 문법 검증을 수행한다.
3. API/Socket 스펙 변경이 있으면 `docs/common/api-spec.md`, `docs/common/socket-events.md`를 같은 작업에서 갱신한다.
4. 인증/환경변수 변경이 있으면 `.env` 키 목록과 `docs/common/architecture.md`를 함께 점검한다.
5. 운영 장애 대응을 위해 클라이언트 노출 메시지와 서버 내부 로그를 구분한다.
6. 문서 업데이트 중 코드 이슈(빌드 실패 등)를 발견하면 문서 PR과 별도로 이슈를 분리해 추적한다.

## 운영 문서 연계 규칙

1. 배포 관련 변경은 `docs/operations/deploy-runbook.md` 체크리스트를 기준으로 검증한다.
2. 장애/운영 이슈 기준은 `docs/operations/ops-log-policy.md`를 따른다.
3. 장애 발생 후에는 원인/조치/재발방지를 문서로 남기고 WBS를 동기화한다.

## PR 공유 규칙

1. 커밋과 푸시가 모두 끝난 상태에서만 PR 정보(링크/요약)를 전달한다.
2. 브랜치 푸시 후에는 PR 생성 링크 또는 Compare 링크를 반드시 함께 전달한다.
3. 사용자에게 공유하는 PR 요약은 아래 항목 순서를 유지한다.
4. 항목명은 임의로 바꾸지 않는다.

```md
Title
`<commit or PR title>`

Summary
<변경 목적과 핵심 요약>

Changed Files
- <file path>

What’s Included
- <구현/수정 내용>

Impact
- <영향 범위>

Validation
- <테스트/검증 결과>

Branch / Commit
- Branch: `<branch-name>`
- Commit: `<commit-sha>`
- Push: `<remote status>`

PR
- <pull request url or compare url>
```
